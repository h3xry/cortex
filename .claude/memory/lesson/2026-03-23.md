# Lessons: 2026-03-23

## From CLI Launcher MVP session

- **Vite HMR intercepts /ws WebSocket paths**: Vite's dev server handles WebSocket upgrade requests for paths starting with `/ws` before the proxy can forward them. Use a different prefix like `/stream` or connect directly to the backend port in dev mode.

- **capture-pane ≠ pipe-pane for TUI apps**: `tmux capture-pane` returns a plain text snapshot of the visible pane (2D grid). For TUI apps like Claude Code that use cursor movements, colors, and box drawing, use `tmux pipe-pane` which captures the raw PTY output stream with all ANSI escape sequences intact.

- **Test full user flow before commit**: Launching Claude CLI requires accepting a trust dialog interactively. This was not caught during implementation because no manual smoke test was done. Always test the complete flow (launch → interact → verify output) before committing.

- **Add security after basic flow works**: Adding CORS restrictions and WebSocket origin checks before the basic WebSocket connection was proven working made debugging harder — couldn't distinguish auth failures from connection failures. Get the happy path working first, then layer security on top.

- **Browser WebSocket origin behavior**: When a browser at `localhost:5173` connects to `ws://localhost:3001`, it sends `Origin: http://localhost:5173`. Server-side origin validation must accept the browser's origin, not just the server's own origin. This is different from HTTP CORS which is enforced by the browser.

- **Follow your own research decisions**: If the research phase chose pipe-pane, implement pipe-pane — not the "easier" capture-pane. Taking shortcuts during implementation leads to rework later.

## From Session Input & Tool Selector session

- **Keyboard event routing needs explicit rules**: When a web input box coexists with terminal control keys, use modifier keys (Alt+Arrow) for terminal controls and leave unmodified keys for the input box. Don't intercept keys like Backspace/Arrow that are needed for normal text editing.

- **Update mocks when adding module exports**: When adding new functions to a module that is mocked in tests, grep for all test files that mock that module and add the new exports to every mock. Missing mock exports cause cryptic "No export defined on mock" errors.
